# 006 - 截图账单 OCR 导入

## 目标

新增根据微信、支付宝、美团月付长截图识别流水候选记录，并在用户预览确认后复用现有批量导入接口保存数据。

## 技术路径

- Cashbook 主应用不直接依赖 PaddleOCR 或 ONNX Runtime。
- 新增独立 `ocr-service`，通过 HTTP 提供 OCR 能力。
- 主应用新增 `/api/entry/flow/screenshot/recognize`，上传图片到 OCR 服务，拿到带坐标文本块后按来源规则解析为 `Flow[]` 候选。
- 用户在流水页选择“截图账单”，手动选择来源、年份、图片，识别后预览并确认导入。

## 配置

主应用环境变量：

```env
NUXT_OCR_SERVICE_URL="http://ocr:8000/ocr"
NUXT_OCR_REQUEST_TIMEOUT_MS="60000"
```

未配置 OCR 服务时，截图识别入口会请求失败并提示配置服务地址。

## OCR 服务部署

本地开发建议先单独启动 OCR 服务，再启动 Cashbook。Cashbook 和 OCR 服务通过 HTTP 通信，不需要把 OCR 依赖安装到主项目里。

OCR 服务按 Python 3.12 维护：

- CPU 镜像：`paddleocr==3.7.0` + `paddlepaddle==3.2.2`，默认 `turbo/tiny`。
- GPU 镜像：NVIDIA CUDA 12.6/cuDNN runtime + `paddleocr==3.7.0` + `onnxruntime-gpu==1.20.1`，默认 `OCR_ENGINE=onnxruntime`、`OCR_DEVICE=gpu`、`fast/small`。

GPU 镜像不再安装 `paddlepaddle-gpu`。PaddleOCR/PaddleX 仍负责 OCR 流水线，真正模型推理由 ONNX Runtime CUDA Execution Provider 执行。

### 1. 启动 OCR 服务

Docker CPU：

```powershell
cd ocr-service
.\scripts\build-cpu.ps1
docker compose -f compose.cpu.yaml up -d
```

Docker GPU：

```powershell
cd ocr-service
.\scripts\build-gpu.ps1
docker compose -f compose.gpu-cu126.yaml up -d
```

GPU Compose 文件使用 `gpus: all` 申请 GPU，并设置 `NVIDIA_VISIBLE_DEVICES=all`、`NVIDIA_DRIVER_CAPABILITIES=compute,utility`。只设置 NVIDIA 环境变量不会让 Docker 自动分配 GPU。

GPU 镜像默认从 `nvidia/cuda:12.6.3-cudnn-runtime-ubuntu24.04` 起步，直接复用 NVIDIA 官方 CUDA/cuDNN runtime 层。APT 只安装 Python 和基础系统库，默认使用国内源，Debian/Ubuntu 包走阿里云镜像。

如需切换国内镜像源：

```powershell
.\scripts\build-gpu.ps1 `
  -AptDebianMirror http://mirrors.tuna.tsinghua.edu.cn/debian `
  -AptDebianSecurityMirror http://mirrors.tuna.tsinghua.edu.cn/debian-security `
  -AptUbuntuMirror http://mirrors.tuna.tsinghua.edu.cn/ubuntu
```

构建时预下载模型：

```powershell
.\scripts\build-cpu.ps1 -PreloadModels
.\scripts\build-gpu.ps1 -PreloadModels
```

模型预热会写入镜像内 `/opt/paddlex-cache`，容器启动时会合并到运行时缓存目录。

### 2. 模型持久化

`compose.cpu.yaml` 会把模型缓存挂载到宿主机目录；`compose.gpu-cu126.yaml` 使用 Docker named volume `paddlex-cache` 持久化模型：

```text
CPU: ocr-service/.paddlex-cache -> /app/.paddlex-cache
GPU: paddlex-cache -> /app/.paddlex-cache
```

只要不删除 `ocr-service/.paddlex-cache` 或 GPU named volume，重建镜像、重建容器后都不会重复下载已缓存模型。

GPU ONNX Runtime 会下载 ONNX 格式模型，缓存目录通常是：

```text
ocr-service/.paddlex-cache/official_models/PP-OCRv6_small_det_onnx
ocr-service/.paddlex-cache/official_models/PP-OCRv6_small_rec_onnx
```

旧的 Paddle 静态模型缓存，如 `PP-OCRv6_small_det`、`PP-OCRv6_small_rec`，不能直接给 ONNX Runtime 使用，因为目录里没有 `inference.onnx`。

如需固定模型目录，可设置：

```env
OCR_TEXT_DETECTION_MODEL_DIR="/app/.paddlex-cache/official_models/PP-OCRv6_small_det_onnx"
OCR_TEXT_RECOGNITION_MODEL_DIR="/app/.paddlex-cache/official_models/PP-OCRv6_small_rec_onnx"
```

### 3. GPU 依赖策略

GPU 镜像默认基础镜像：

- `nvidia/cuda:12.6.3-cudnn-runtime-ubuntu24.04`

GPU 镜像默认 apt 包：

- `python3`
- `python3-venv`
- `ca-certificates`
- `libgomp1`
- `libgl1`
- `libglib2.0-0`
- `zlib1g`

不要把宿主机 `/usr/local/cuda` 挂进容器。宿主机只安装 NVIDIA 驱动和 NVIDIA Container Toolkit，容器内复用官方 CUDA/cuDNN runtime 基础镜像，运行时只暴露 `compute,utility` 驱动能力。

重复构建时，Docker 会复用 NVIDIA 基础镜像层；`Dockerfile.gpu` 使用 BuildKit cache mount 缓存 `/var/cache/apt` 和 `/var/lib/apt/lists`；pip 继续缓存到 `/tmp/pip-cache`。

### 4. 健康检查

```powershell
Invoke-RestMethod http://127.0.0.1:8000/health
```

CPU 镜像返回中应包含：

```json
{
  "ok": true,
  "device": "cpu",
  "performanceMode": "turbo",
  "engine": null
}
```

GPU 镜像返回中应包含：

```json
{
  "ok": true,
  "device": "gpu",
  "performanceMode": "fast",
  "engine": "onnxruntime",
  "onnxRuntimeProviders": ["CUDAExecutionProvider", "CPUExecutionProvider"]
}
```

如果启动时报 `NVIDIA driver runtime is not visible`，或 Docker 报 `no known GPU vendor found`，说明容器没有拿到 GPU，优先检查 Docker Desktop/WSL GPU 支持、NVIDIA 驱动和 NVIDIA Container Toolkit。如果 `onnxRuntimeProviders` 中没有 `CUDAExecutionProvider`，说明 GPU 版 ONNX Runtime 未加载 CUDA provider。

也可以用一张截图直接测试 OCR：

```powershell
curl.exe -F "image=@C:\path\to\bill.jpg" http://127.0.0.1:8000/ocr
```

## 本地 Python 启动

CPU：

```powershell
cd ocr-service
py -3.12 -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install --upgrade pip
python -m pip install -r requirements.txt
uvicorn app:app --host 0.0.0.0 --port 8000
```

GPU/ONNX Runtime：

```powershell
cd ocr-service
py -3.12 -m venv .venv-gpu
.\.venv-gpu\Scripts\Activate.ps1
python -m pip install --upgrade pip
python -m pip install -r requirements-gpu.txt
$env:OCR_DEVICE="gpu"
$env:OCR_ENGINE="onnxruntime"
$env:OCR_PERFORMANCE_MODE="fast"
uvicorn app:app --host 0.0.0.0 --port 8000
```

Windows 本地 GPU 运行需要系统 PATH 能找到 CUDA/cuDNN 动态库；日常部署更推荐 Docker。

## Cashbook 本地环境

如果本地还没有 `.env`，先复制示例配置：

```powershell
Copy-Item env .env
```

在 `.env` 中追加：

```env
NUXT_OCR_SERVICE_URL="http://127.0.0.1:8000/ocr"
NUXT_OCR_REQUEST_TIMEOUT_MS="120000"
```

修改 `.env` 后需要重启 Cashbook dev server：

```powershell
npm run dev
```

然后进入流水页：`导入导出 -> 截图账单`，选择来源、年份和截图，点击识别。

## 常见部署 URL

| 场景 | `NUXT_OCR_SERVICE_URL` |
|------|-------------------------|
| Cashbook 本机开发，OCR 本机/Docker 暴露 8000 | `http://127.0.0.1:8000/ocr` |
| Cashbook 与 OCR 在同一个 Compose 网络 | `http://ocr:8000/ocr` |
| Cashbook 容器访问宿主机 OCR，Docker Desktop | `http://host.docker.internal:8000/ocr` |

## OCR 服务常用环境变量

| 变量 | CPU 默认 | GPU 默认 | 说明 |
|------|----------|----------|------|
| `OCR_DEVICE` | `cpu` | `gpu` | PaddleOCR 设备参数 |
| `OCR_ENGINE` | 空 | `onnxruntime` | PaddleOCR/PaddleX 推理引擎 |
| `OCR_PERFORMANCE_MODE` | `turbo` | `fast` | `turbo=tiny`，`fast=small`，`accurate=medium` |
| `OCR_ENABLE_MKLDNN` | `true` | `false` | CPU oneDNN/MKLDNN 加速；GPU 镜像默认关闭 |
| `OCR_CPU_THREADS` | `4` | `2` | CPU 推理线程数 |
| `OCR_REC_BATCH_SIZE` | `16` | `32` | 文本识别 batch size |
| `OCR_MODEL_SOURCE` | `bos` | `bos` | PaddleX 模型下载源 |
| `OCR_CACHE_HOME` | `/app/.paddlex-cache` | `/app/.paddlex-cache` | 模型缓存目录 |
| `OCR_TEMP_DIR` | `/app/.paddlex-cache/tmp` | `/app/.paddlex-cache/tmp` | 模型下载/解压临时目录 |
| `OCR_TEXT_DETECTION_MODEL_DIR` | 空 | 空 | 可选，显式指定检测 ONNX 模型目录 |
| `OCR_TEXT_RECOGNITION_MODEL_DIR` | 空 | 空 | 可选，显式指定识别 ONNX 模型目录 |
| `OCR_ENGINE_CONFIG` | 空 | 空 | 可选，JSON 对象，透传给 PaddleOCR `engine_config` |
| `OCR_MAX_SINGLE_IMAGE_HEIGHT` | `6000` | `6000` | 超过后按长图切片 |
| `OCR_SLICE_HEIGHT` | `6000` | `6000` | 长图切片高度 |
| `OCR_SLICE_OVERLAP` | `320` | `320` | 切片重叠高度 |

`OCR_MODEL_SOURCE` 会映射到 PaddleX 的 `PADDLE_PDX_MODEL_SOURCE`。默认使用 `bos`，避免 Windows 本地默认 HuggingFace snapshot 缓存时偶发的权限/链接问题。可选值包括 `bos`、`huggingface`、`modelscope`、`aistudio`。

`OCR_PERFORMANCE_MODE` 用于选择 PP-OCRv6 模型大小：`turbo` 使用 tiny，`fast` 使用 small，`accurate` 使用 medium。CPU 默认 `turbo`，GPU 默认 `fast`。

`OCR_TEMP_DIR` 默认是 `.paddlex-cache/tmp`。Windows 本地建议保留该配置，避免系统临时目录权限异常导致模型下载失败。

## 规则配置

默认规则位于 `server/utils/screenshotOcr/rules.ts`，后台“系统设置 / OCR规则”可编辑覆盖规则。覆盖规则保存到 `NUXT_DATA_PATH/ocr-rules.json`，不改数据库结构。

规则关键字段：

- `amountArea`：金额列横向范围，按截图宽度百分比。
- `titleArea`：标题列横向范围，按截图宽度百分比。
- `datePatterns`：流水时间正则。
- `monthPatterns` / `simpleMonthPatterns`：月份分组正则。
- `incomeKeywords` / `expenseKeywords` / `neutralKeywords`：无符号金额的收支推断关键词。
- `minConfidence`：低置信度提示阈值。

## OCR 长截图处理

`ocr-service` 对 6000px 内的截图默认整图识别，超过后按高度分片识别，默认每片 6000px、重叠 320px。每个分片的 OCR 坐标会加回全图 `offsetY`，重叠区按相同文本和相近坐标去重。

## 导入策略

识别结果只作为候选流水，不直接入库。用户可在预览中修正日期、收支、金额、名称、类型，确认后仍调用 `/api/entry/flow/imports`。
